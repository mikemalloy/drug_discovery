# Drug Discovery Platform вҖ” Frontend Design Spec
**Date:** 2026-06-02
**Status:** Approved
**Phase:** 4 вҖ” Frontend

---

## Purpose

A single-page Next.js UI for the drug discovery platform. The user enters a SMILES string, clicks Analyze, and sees a structured safety and toxicity report populated in two result cards. Intended as a portfolio demo вҖ” simple first pass, designed to be handed off to V0 for visual elaboration.

---

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Layout | Left input panel + right stacked result cards | Persistent input lets users run multiple compounds without scrolling |
| Theme | Light / clinical (white + blue) | Feels like a pharma SaaS product, professional |
| Loading state | Progress steps checklist, response-driven | Makes 60вҖ“120s wait feel purposeful; skips steps if API responds early |
| Structure SVG | Rendered from `report.structure_svg` via `dangerouslySetInnerHTML` | Already generated by RDKit in backend, zero extra dependencies |

### Progress Steps

Four steps (in order):
1. `Validating SMILES`
2. `Computing toxicity scores`
3. `Computing explainability`
4. `Generating report`

**Timer schedule** (fires only if fetch has not yet resolved):
- Step 1 вҶ’ Step 2: 5 seconds
- Step 2 вҶ’ Step 3: 15 seconds
- Step 3 вҶ’ Step 4: 45 seconds

**On Analyze click:**
1. Validate SMILES is non-empty (button is disabled otherwise вҖ” client guard only)
2. Set `status = 'loading'`, `stepIndex = 0`, `report = null`, `errorMessage = ''`
3. Start three `setTimeout` calls (stored in refs for cleanup)
4. Fire `fetch(NEXT_PUBLIC_API_URL/analyze, { method: 'POST', body: { smiles, compound_name } })`
5. When fetch settles: clear all pending timers
   - Success (200): set `status = 'done'`, `report = data`
   - 422: set `status = 'error'`, `errorMessage = 'Invalid SMILES string вҖ” please check your input'`
   - Other error / network failure: set `status = 'error'`, `errorMessage = 'Could not reach the analysis server'`

Both cards read the same `stepIndex` вҖ” they stay in sync.

## Error Handling Summary

| Scenario | Trigger | Display |
|----------|---------|---------|
| Empty SMILES | Client-side | Analyze button disabled |
| Invalid SMILES | 422 from API | "Invalid SMILES string вҖ” please check your input" in both cards |
| Network / server error | fetch throws or non-200/422 | "Could not reach the analysis server" + Retry button |
| Retry | User clicks Retry | Resets `status` to `'idle'`, clears `report` and `errorMessage` |
